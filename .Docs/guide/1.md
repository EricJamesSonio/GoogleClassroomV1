## 1. Project Overview

### What This App Does

This is a simplified Google Classroom clone focused on:

- **Authentication** — Register and login as educator or student
- **Classes** — Educators create classes, students join via invite or request
- **Meetings** — Scheduled video meetings with countdown timer and auto-start
- **Video Calls** — Real-time video using Agora RTC SDK
- **Live Chat** — In-meeting chat via WebSockets
- **Recording** — Optional meeting recording via Agora Cloud Recording
- **File Storage** — Cloudinary for any media/file uploads

### What This App Does NOT Do (Intentionally Scoped Out)

- Assignments (handled in separate repo)
- Announcements
- Push/device notifications
- Complex grading systems

```

      |

      |
FastAPI REST API ←→ FastAPI WebSocket Server
      |
Beanie ODM
      |
MongoDB Atlas (or local)

      Separately:
React ←→ Agora SDK (video/audio peer-to-peer)
FastAPI → Agora REST API (token generation, cloud recording)
React → Cloudinary SDK (file uploads)
```

### Request Flow Example — Joining a Meeting

```
1. Student opens meeting page
2. React calls GET /meetings/{id} via Axios
3. FastAPI returns meeting details + Agora channel name
4. React calls GET /meetings/{id}/agora-token
5. FastAPI generates Agora token server-side, returns it
6. React initializes Agora client with channel + token
7. Student joins video call
8. React opens WebSocket to /ws/meetings/{id}
9. Live chat messages flow through WebSocket
```

## 4. Database Schema Design

MongoDB uses collections instead of tables. This app uses **7 collections**.

---

### Collection 1: `users`

```json
{
  "_id": "ObjectId",
  "name": "Juan Dela Cruz",
  "email": "juan@email.com",
  "password_hash": "bcrypt_hashed",
  "role": "student | educator",
  "avatar_url": "https://cloudinary.com/...",
  "created_at": "ISODate"
}
```

**Why this design:**
- `role` is the key field — it gates all permissions throughout the app
- `avatar_url` stored as Cloudinary URL, not binary data
- Password is always stored hashed (bcrypt via passlib)

---

### Collection 2: `classes`

```json
{
  "_id": "ObjectId",
  "name": "Software Engineering 101",
  "description": "Introduction to SE concepts",
  "educator_id": "ObjectId → users",
  "cover_image_url": "https://cloudinary.com/...",
  "created_at": "ISODate"
}
```

**Why this design:**
- Simple. One educator owns one class.
- `educator_id` is a reference to `users` — not embedded, because users are queried independently
- Cover image stored as Cloudinary URL

---

### Collection 3: `class_members`

```json
{
  "_id": "ObjectId",
  "class_id": "ObjectId → classes",
  "user_id": "ObjectId → users",
  "role": "student | educator",
  "joined_at": "ISODate"
}
```

**Why this design:**
- Handles the many-to-many relationship between users and classes
- A class has many students; a student can be in many classes
- Querying all members of a class = `find({ class_id: X })`
- Querying all classes of a student = `find({ user_id: X })`
- The educator is also inserted here when they create the class, so one query gets all participants

---

### Collection 4: `class_invitations`

```json
{
  "_id": "ObjectId",
  "class_id": "ObjectId → classes",
  "student_id": "ObjectId → users",
  "invited_by": "ObjectId → users (educator)",
  "status": "pending | accepted | rejected",
  "created_at": "ISODate"
}
```

**Why this design:**
- Separates the invitation lifecycle from actual membership
- Students are only added to `class_members` AFTER accepting
- `status` field drives the flow: pending → accepted → triggers insert to class_members
- Supports both: educator invites student, or student requests to join (invited_by = student themselves, status starts as "requested")

**Invite Flow:**
```
Educator invites → status: "pending"
Student accepts  → status: "accepted" + insert to class_members
Student rejects  → status: "rejected"

Student requests → status: "requested" (invited_by = student_id)
Educator accepts → status: "accepted" + insert to class_members
Educator rejects → status: "rejected"
```

---

### Collection 5: `meetings`

```json
{
  "_id": "ObjectId",
  "class_id": "ObjectId → classes",
  "created_by": "ObjectId → users (educator)",
  "title": "Lecture 3: Async Programming",
  "status": "scheduled | live | ended",
  "scheduled_at": "ISODate",
  "started_at": "ISODate | null",
  "ended_at": "ISODate | null",
  "agora_channel": "unique_channel_name_string",
  "recording_url": "https://cloudinary.com/... | null",
  "created_at": "ISODate"
}
```

**Why this design:**
- `scheduled_at` is the single source of truth for when the meeting starts
- `agora_channel` is generated at meeting creation — a unique string used to join the Agora channel. Store it here so all participants use the same channel.
- `status` is managed by a background scheduler (APScheduler), not manually by educator
- `recording_url` is populated after recording ends and is processed by Agora Cloud Recording callback

**Meeting Timer Design:**

When the educator creates a meeting, they pick one of:
- A specific datetime (e.g., "March 10, 2026 at 2:00 PM")
- A relative time (e.g., "Start in 30 minutes") — converted to absolute datetime on the server

The frontend calculates the countdown using `scheduled_at - now`. When countdown hits zero, the status transitions to `live` and the Agora channel becomes joinable.

---

### Collection 6: `meeting_invitations`

```json
{
  "_id": "ObjectId",
  "meeting_id": "ObjectId → meetings",
  "user_id": "ObjectId → users",
  "invited_by": "ObjectId → users",
  "status": "invited | requested | accepted | rejected",
  "created_at": "ISODate"
}
```

**Why this design:**
- Mirrors class_invitations logic but scoped to a single meeting
- Supports all four scenarios:
  1. Invite all class members → bulk insert with status "invited"
  2. Invite selected students → individual inserts
  3. Student not in class requests to join → status "requested"
  4. Educator accept/reject requests

---

### Collection 7: `meeting_messages`

```json
{
  "_id": "ObjectId",
  "meeting_id": "ObjectId → meetings",
  "sender_id": "ObjectId → users",
  "sender_name": "Juan Dela Cruz",
  "message": "Can you repeat that?",
  "created_at": "ISODate"
}
```

**Why this design:**
- Chat messages are simple — no threading, no reactions (for now)
- `sender_name` is denormalized (stored directly) so chat history renders without extra user lookups
- `created_at` sorts messages chronologically
- Messages are sent via WebSocket in real-time AND persisted here so users can scroll up history

---

### Schema Relationships Summary

```
users
  └─ creates ──────────────→ classes
  └─ joins via ────────────→ class_members (many-to-many)
  └─ invited via ──────────→ class_invitations

classes
  └─ has many ─────────────→ meetings

meetings
  └─ has many ─────────────→ meeting_invitations
  └─ has many ─────────────→ meeting_messages
```

## 6. API Design

```

GET    /auth/me                Get current user from JWT
```

### Classes

```

GET    /classes/{class_id}               Get class details + member list
```

### Class Invitations

```
POST   /classes/{class_id}/invite        Educator: invite a student by email
GET    /invitations                      Student: get all my pending invitations
POST   /invitations/{id}/accept          Student: accept invitation
POST   /invitations/{id}/reject          Student: reject invitation
POST   /classes/{class_id}/request-join  Student: request to join a class
GET    /classes/{class_id}/join-requests Educator: see all join requests

### Meetings

```
POST   /classes/{class_id}/meetings      Educator: create meeting with scheduled_at

PATCH  /meetings/{meeting_id}/end        Educator: manually end meeting
```

### Meeting Invitations

```
POST   /meetings/{id}/invite-all         Educator: invite all class members
POST   /meetings/{id}/invite             Educator: invite selected students

## 7. Step-by-Step Build Plan

Build the project in phases. Each phase is a working, testable increment.

---

**Goal:** Running FastAPI server connected to MongoDB.

Tasks:
- Create FastAPI project structure (folders as in Section 5)
- Setup `.env` with all keys (MONGO_URI, JWT_SECRET, AGORA_APP_ID, AGORA_APP_CERT, CLOUDINARY_URL)
- Connect MongoDB via Motor + Beanie in `database.py`
- Define all 7 Beanie Document models
- Test connection with a health check endpoint `GET /health`

**Install:**
```bash
pip install fastapi uvicorn beanie motor python-jose passlib bcrypt cloudinary agora-token-builder apscheduler python-dotenv
```

---

### Phase 2 — Authentication

**Goal:** Users can register and login. JWT protects routes.

Tasks:
- `POST /auth/register` — hash password, save user, return token
- `POST /auth/login` — verify password, return JWT
- `GET /auth/me` — decode JWT, return user
- Create `get_current_user` dependency used in all protected routes

**Test:** Register as educator, register as student, login both, call /auth/me.

---

### Phase 3 — Classes

**Goal:** Educators can create and view classes.

Tasks:
- `POST /classes` — educator only, creates class, inserts educator into class_members
- `GET /classes` — returns classes where current user is a member
- `GET /classes/{id}` — returns class + member list

**Test:** Create a class as educator, verify it shows in GET /classes.

---

### Phase 4 — Class Invitations & Join Requests

**Goal:** Students join classes through invite flow or request flow.

Tasks:
- `POST /classes/{id}/invite` — educator sends invite, creates class_invitation with status "pending"
- `GET /invitations` — student sees their pending invites
- `POST /invitations/{id}/accept` — student accepts, triggers insert to class_members
- `POST /invitations/{id}/reject` — student rejects
- `POST /classes/{id}/request-join` — student requests, creates class_invitation with status "requested"
- Educator can view and accept/reject join requests

**Test:** Full invite flow. Full request-to-join flow.

---

### Phase 5 — Meetings (Scheduling + Auto-Start)

**Goal:** Educators schedule meetings. Countdown shows on frontend. Meeting auto-starts on time.

Tasks:
- `POST /classes/{id}/meetings` — accepts `scheduled_at` (ISO datetime) OR `start_in_minutes` (converted to datetime server-side). Generates unique `agora_channel` string. Saves meeting with status "scheduled".
- `GET /meetings/{id}` — returns meeting data including `scheduled_at` so frontend can compute countdown
- Setup APScheduler in `main.py` lifespan — schedule a job for each meeting's `scheduled_at` that flips status to "live"
- `PATCH /meetings/{id}/end` — educator manually ends, flips status to "ended"

**Test:** Create meeting in 2 minutes. Watch status flip to "live" automatically.

---

### Phase 6 — Meeting Invitations

**Goal:** Educator controls who is invited to a meeting.

Tasks:
- `POST /meetings/{id}/invite-all` — bulk inserts all class_members as meeting_invitations
- `POST /meetings/{id}/invite` — invite specific students
- Student can request to join a meeting they weren't invited to
- Educator accept/reject requests

**Test:** Create meeting, invite all, verify invitations created.

---

### Phase 7 — Agora Token Generation

**Goal:** Frontend can get a valid Agora token to join the video channel.

Tasks:
- Install `agora-token` (Python Agora token builder)
- `GET /meetings/{id}/agora-token` — generate RTC token using AGORA_APP_ID + AGORA_APP_CERT + channel name + user uid. Return token + channel name + app id.
- Token should only be generated if meeting status is "live" and user is invited/accepted

**Test:** Get token, verify format is correct. Use Agora test console to verify token works.

---

### Phase 8 — Live Chat (WebSocket)

**Goal:** Participants send and receive chat messages in real time during a meeting.

Tasks:
- Build `ConnectionManager` in `websocket/meeting_chat.py` — manages active connections per meeting
- `WS /ws/meetings/{meeting_id}` — on connect: authenticate via JWT query param, load message history, broadcast new messages, remove on disconnect
- `GET /meetings/{id}/messages` — return paginated historical messages
- Persist every message to `meeting_messages` collection

**Test:** Open two browser tabs, send messages from each, verify both receive.

---

### Phase 9 — Video Meeting (Agora RTC Frontend)

**Goal:** Users join a video call inside the meeting room.

Tasks:
- Install Agora Web SDK in React: `npm install agora-rtc-react`
- On meeting page load: call `GET /meetings/{id}/agora-token`
- Initialize Agora client, join channel with token
- Render local video + remote participant videos
- Mic/camera toggle buttons
- Leave meeting button

**Test:** Two users join same meeting. Verify audio and video work.

### Phase 10 — Recording

**Goal:** Educator can start and stop recording a meeting.

Tasks:

## 8. Key Business Logic

### Role Guard (Dependency)

Every route that is educator-only uses a FastAPI dependency:

```python
def require_educator(current_user = Depends(get_current_user)):
    if current_user.role != "educator":
        raise HTTPException(status_code=403, detail="Educator access only")
    return current_user
```

### Invite vs Request Flow

Both use the same `class_invitations` collection. The difference is who initiates:

- **Educator invites:** `invited_by = educator_id`, `student_id = target`, `status = "pending"`
- **Student requests:** `invited_by = student_id`, `student_id = student_id`, `status = "requested"`

When accepted (by either party), a `class_members` document is inserted.

### Preventing Duplicate Members

Before inserting to `class_members`, always check:

```python
existing = await ClassMember.find_one({"class_id": class_id, "user_id": user_id})
if existing:
    return  # already a member, skip insert
```

### Meeting Channel Name

Generate at meeting creation time. Must be unique. Use a combination of class ID + timestamp or UUID:

```python
import uuid
agora_channel = f"meeting_{str(uuid.uuid4()).replace('-', '')[:16]}"
```

Store this in the meeting document. All token generation uses this same channel name.

## 9. Meeting Timer & Auto-Start Logic

### Creating a Meeting

The educator can provide either:

```json
{ "scheduled_at": "2026-03-10T14:00:00Z" }
```

or:

```json
{ "start_in_minutes": 30 }
```

The backend converts `start_in_minutes` to an absolute datetime:

```python
from datetime import datetime, timedelta, timezone

if start_in_minutes:
    scheduled_at = datetime.now(timezone.utc) + timedelta(minutes=start_in_minutes)
```

### Scheduling the Auto-Start Job

When a meeting is created, register a job with APScheduler:

```python
from apscheduler.schedulers.asyncio import AsyncIOScheduler

scheduler = AsyncIOScheduler()

async def start_meeting(meeting_id: str):
    meeting = await Meeting.get(meeting_id)
    meeting.status = "live"
    meeting.started_at = datetime.now(timezone.utc)
    await meeting.save()

scheduler.add_job(
    start_meeting,
    trigger="date",
    run_date=scheduled_at,
    args=[str(meeting.id)]
)
```

Start scheduler in FastAPI lifespan:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    scheduler.start()
    yield
    scheduler.shutdown()
```

### Frontend Countdown

The React meeting page receives `scheduled_at` from the API. It computes and displays the countdown using `setInterval`:

```javascript
const diff = new Date(scheduledAt) - new Date();
// display hours, minutes, seconds
```

When `diff <= 0`, the frontend polls `GET /meetings/{id}` every 5 seconds until `status === "live"`, then renders the Agora video room automatically.

## 10. Agora Video Integration

### Flow Summary

```
1. Meeting created → agora_channel stored in DB
2. User opens meeting page → GET /meetings/{id}/agora-token
3. Backend generates token using Agora Token Builder
4. Frontend joins Agora channel with token
5. Peer-to-peer audio/video via Agora infrastructure
```

### Backend: Token Generation

```python
from agora_token_builder import RtcTokenBuilder, Role_Publisher
import time

def generate_agora_token(channel_name: str, uid: int) -> str:
    expiry = int(time.time()) + 3600  # 1 hour
    return RtcTokenBuilder.buildTokenWithUid(
        AGORA_APP_ID,
        AGORA_APP_CERT,
        channel_name,
        uid,
        Role_Publisher,
        expiry
    )
```

The `uid` can be derived from the user's ObjectId converted to an integer, or use `0` for dynamic assignment.

### Frontend: Joining the Channel

```javascript
import AgoraRTC from "agora-rtc-react";

const client = AgoraRTC.createClient({ mode: "rtc", codec: "vp8" });

// On meeting page mount:
await client.join(AGORA_APP_ID, channelName, token, uid);
const localTrack = await AgoraRTC.createMicrophoneAndCameraTracks();
await client.publish(localTrack);
```

### Recording

Recording is triggered server-side via Agora Cloud Recording REST API. Three steps:
1. **Acquire** — get a resourceId
2. **Start** — begin recording with cloud storage config
3. **Stop** — end recording, Agora uploads file to your storage

Store `resourceId` and `sid` on the meeting document during recording so they can be used for the stop call.

## 11. WebSocket Live Chat

### Connection Manager

```python
class ConnectionManager:
    def __init__(self):
        self.active: dict[str, list[WebSocket]] = {}

    async def connect(self, meeting_id: str, ws: WebSocket):
        await ws.accept()
        self.active.setdefault(meeting_id, []).append(ws)

    def disconnect(self, meeting_id: str, ws: WebSocket):
        self.active[meeting_id].remove(ws)

    async def broadcast(self, meeting_id: str, message: dict):
        for ws in self.active.get(meeting_id, []):
            await ws.send_json(message)
```

### WebSocket Endpoint

```python
@router.websocket("/ws/meetings/{meeting_id}")
async def meeting_chat(websocket: WebSocket, meeting_id: str, token: str = Query(...)):
    user = verify_jwt(token)  # authenticate via query param
    await manager.connect(meeting_id, websocket)
    try:
        while True:
            data = await websocket.receive_json()
            msg = await MeetingMessage(
                meeting_id=meeting_id,
                sender_id=user.id,
                sender_name=user.name,
                message=data["message"]
            ).insert()
            await manager.broadcast(meeting_id, {
                "sender": user.name,
                "message": data["message"],
                "created_at": str(msg.created_at)
            })
    except WebSocketDisconnect:
        manager.disconnect(meeting_id, websocket)
```

### Frontend Connection

```javascript
const ws = new WebSocket(`ws://localhost:8000/ws/meetings/${meetingId}?token=${jwt}`);
ws.onmessage = (event) => {
    const msg = JSON.parse(event.data);
    setMessages(prev => [...prev, msg]);
};
ws.send(JSON.stringify({ message: inputText }));
```
